# 球探浏览器自动抓取预测设计

## 1. 背景

当前工程已经具备以下能力：

- 使用 `football-data.co.uk` 完成历史训练集构建
- 训练冷门筛选基线模型并输出 `Top-N` 指标
- 通过手工快照 CSV 运行无 API Key 的预测流程

离最终目标还差一段关键链路：

- 自动获取未来比赛的赔率数据
- 自动标准化为模型输入
- 自动输出逐场概率与冷门排行榜

由于当前没有可用的商业赔率 API Key，本子项目选择浏览器自动化方案，直接基于球探公开页面完成“抓取 -> 解析 -> 预测”闭环。

## 2. 子项目目标

本阶段只解决“未来比赛自动预测”，不做自动重训。

交付目标：

- 输入比赛日期范围
- 自动打开球探公开页面并抓取比赛列表
- 自动抓取每场比赛的 `1X2`、亚盘、大小球页面
- 解析为现有模型可消费的快照字段
- 自动输出逐场冷门概率、`Top-N` 冷门榜和失败报告

## 3. 范围与非目标

### 3.1 范围

- 数据来源：球探公开网页
- 技术方式：真实浏览器会话自动化，不使用命令行直连抓站
- 预测任务：`home_upset_win`、`away_upset_win`、`non_upset`
- 输出形式：JSON/CSV 预测结果 + 原始 HTML 缓存 + 失败日志

### 3.2 非目标

- 不在本阶段实现自动重训
- 不引入新的实时 API 供应商
- 不在本阶段扩展平局冷门任务
- 不重写现有训练主干
- 不做完整盘口时间序列数据库

## 4. 方案选择

本子项目采用：

- `球探公开页 + 浏览器自动化 + 原始缓存 + 解析器`

不采用以下方案作为本阶段主路径：

- `API 方案`
  原因：当前没有可用 Key，无法满足“立即自动化”的要求。
- `第三方赔率站抓取`
  原因：条款风险、页面不稳定和字段口径偏差更高。
- `直接人工复制粘贴`
  原因：只能作为过渡，不满足最终自动预测目标。

## 5. 系统边界

本阶段新增 4 个逻辑模块：

### 5.1 `browser_collector`

负责驱动真实浏览器完成：

- 打开指定日期的赛程页
- 获取比赛列表
- 按比赛进入赔率详情页
- 保存原始页面源码

### 5.2 `raw_cache`

负责缓存运行过程中抓到的页面：

- 日期赛程页 HTML
- 每场比赛的欧赔页 HTML
- 每场比赛的亚盘页 HTML
- 每场比赛的大小球页 HTML

缓存必须按日期和比赛维度组织，保证失败时可复现。

### 5.3 `titan007_parser`

负责把赛程页和赔率页解析成结构化数据：

- 比赛标识、联赛、开赛时间、主客队
- `1X2` 初赔与即时赔率
- 亚盘初盘与即时盘
- 大小球初盘与即时盘
- 数据完整度标记

### 5.4 `prediction_runner`

负责复用现有模型主干：

- 把解析结果转换成标准快照行
- 调用现有预测逻辑
- 输出逐场概率和 `Top-N`

## 6. 抓取流程

完整流程如下：

1. 输入 `start_date` 与 `end_date`
2. 浏览器自动打开对应日期的球探赛程页
3. 解析当天比赛列表，提取比赛链接或比赛 ID
4. 逐场抓取 `欧赔 / 亚盘 / 大小球` 页面
5. 将原始 HTML 落盘缓存
6. 解析得到标准快照字段
7. 调用现有模型完成预测
8. 输出预测结果、失败报告与数据完整度信息

## 7. 数据流设计

### 7.1 输入

- `start_date`
- `end_date`
- `top_n`
- 可选：浏览器等待时间、是否复用缓存、是否限制联赛范围

### 7.2 中间产物

- `data/raw/titan007_browser/<run_id>/...`
  保存原始页面源码
- `data/interim/titan007_browser/<run_id>/...`
  保存比赛列表、结构化快照和失败明细

### 7.3 输出

- 逐场预测 JSON
- 逐场预测 CSV
- `Top-N` 冷门榜
- 失败比赛清单
- 运行摘要

## 8. 解析边界

首版解析器必须优先满足模型当前需要的字段，而不是试图提取页面全部信息。

最低必需字段：

- `competition_code`
- `match_date`
- `kickoff_time`
- `home_team`
- `away_team`
- `open_home_odds`
- `open_draw_odds`
- `open_away_odds`
- `close_home_odds`
- `close_draw_odds`
- `close_away_odds`

推荐字段：

- `open_ah_line`
- `close_ah_line`
- `open_ah_home_odds`
- `open_ah_away_odds`
- `close_ah_home_odds`
- `close_ah_away_odds`
- `open_over25_odds`
- `open_under25_odds`
- `close_over25_odds`
- `close_under25_odds`

如果 `1X2` 核心字段缺失，则该场不进入预测。
如果亚盘或大小球字段缺失，则允许降级预测，但必须标记为 `partial`。

## 9. 失败处理

系统必须显式失败，不允许静默吞错。

### 9.1 赛程页失败

- 整个日期任务标记失败
- 输出失败原因
- 保留已抓到的原始页面和日志

### 9.2 单场赔率页失败

- 该场标记为失败或部分成功
- 写入失败清单
- 不影响其他比赛继续抓取

### 9.3 页面结构变化

- 保存失败 HTML
- 报告具体是哪个解析器失效
- 不直接抛出不可读 traceback 作为最终结果

### 9.4 浏览器会话不可用

- 明确提示用户重新打开 Chrome 和目标页面
- 不继续执行空抓取

## 10. 与现有代码的集成方式

本阶段尽量做小改动，不重构现有训练主干。

建议改动边界：

- `src/upset_model/collectors/`
  新增球探浏览器抓取与页面解析模块
- `src/upset_model/standardize.py`
  新增从浏览器解析结果生成标准快照行的入口
- `scripts/`
  新增按日期范围自动抓取并预测的 CLI
- `tests/`
  新增 HTML 解析与预测串联测试

现有模型训练、阈值选择和无 Key 快照预测逻辑尽量保持不动。

## 11. 验收标准

本阶段完成的判定标准为：

- 输入一个未来日期范围，脚本能自动抓到比赛列表
- 至少大部分比赛能成功产出 `1X2` 快照
- 在亚盘或大小球缺失时，系统能降级预测并标记完整度
- 能直接输出逐场概率和 `Top-N` 冷门榜
- 失败时能明确区分是赛程页失败、单场赔率页失败还是解析器失效

## 12. 风险与缓解

### 风险 1：球探页面结构调整

- 影响：高
- 缓解：保留原始 HTML 缓存，解析器测试使用固定样本片段

### 风险 2：浏览器自动化不稳定

- 影响：中
- 缓解：首版优先使用真实浏览器会话与显式等待，不追求无头稳定性

### 风险 3：未来页字段与历史训练字段口径不完全一致

- 影响：高
- 缓解：解析阶段统一落为当前标准快照字段，缺失时显式降级

### 风险 4：部分比赛盘口页不完整

- 影响：中
- 缓解：允许 `1X2` 必需、亚盘/大小球可选的降级预测策略
